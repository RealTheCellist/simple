Dark Trading UI for Mobile

시장 관찰부터 알림 이력까지 한 흐름으로.

WATCHR Mobile은 단순 시세 앱이 아니라, 종목 추가부터 관찰, 알림, 기록까지를 하나의 짧은 사용자 루프로 묶는 도구입니다. 이 페이지는 작업지시서의 기술 밀도와 모바일 모크업의 감성을 웹용 컨셉 브리프로 재구성한 버전입니다.

Product Promise

이 앱이 실제로 해결하는 것

WATCHR Mobile은 정보를 나열하는 대신 사용자가 무엇을 보고 언제 반응할지를 제품의 기본 구조로 만듭니다.

01 Watchlist

관심 종목을 한 줄 밀도로 압축

가격, 변화율, 스파크라인, 알림 상태를 한 행에서 동시에 읽도록 설계해 빠른 스캔이 가능합니다.

Dense by design
02 Futures

시장 체온을 먼저 보여주는 대시보드

선물과 주요 지수를 크게, 빠르게, 비교 가능하게 보여 주어 시장 방향성을 먼저 읽게 합니다.

Large numbers first
03 Alerts

조건을 세우고 백그라운드로 맡기기

사용자는 목표가와 조건만 입력하면 됩니다. 앱은 쿨다운과 알림 흐름을 관리합니다.

Action over monitoring
04 History

반응한 이유를 나중에도 남기는 기록

언제 어떤 조건에서 울렸는지 남겨, 나중의 판단 재료와 회고 자료로 활용할 수 있습니다.

Traceable signal log
How It Works

추가 → 관찰 → 알림 → 이력

제품의 핵심은 탭 개수가 아니라 사용자 루프의 짧음입니다. 한 번 종목을 추가하면 이후 흐름이 끊기지 않도록 설계합니다.

01

Add

워치리스트에 종목을 추가하고 기본 가격 조회를 시작합니다.

02

Observe

행 단위 정보와 선물 지표를 함께 보며 시장 흐름과 개별 종목 변화를 스캔합니다.

03

Alert

조건 충족 시 로컬 알림을 발생시키고, 같은 조건은 쿨다운으로 과발동을 막습니다.

04

Review

히스토리 탭에서 발동 기록과 시간 정보를 확인해 나중의 판단 자료로 활용합니다.

System Character

모바일 트레이딩 UI를 웹 브리프로 번역

감성은 다크 터미널, 구조는 제품 문서, 읽기 방식은 랜딩 페이지에 가깝게 재구성했습니다.

Tone
기술적이고 차분하지만 살아 있는 시스템처럼 보여야 합니다.
Priority
큰 숫자, 얇은 경계선, 작은 배지, 읽기 쉬운 카드 위계.
Interaction
과한 애니메이션 대신 탭 전환, 접기/펼치기, 미세한 호버만 사용.
Feature Spotlight

4개 탭을 웹에서 읽기 쉬운 구조로 재해석

아래 탭을 눌러 각 화면의 역할을 확인할 수 있습니다. 모바일 모크업과 연결되지만 이 문서만 읽어도 제품의 역할이 전달되도록 정리했습니다.

Watchlist는 가장 짧은 의사결정 화면입니다.

종목 코드, 가격, 변화율, 미니 추세선, 알림 존재 여부를 한 줄에 모읍니다. 사용자는 스크롤만으로 여러 종목을 빠르게 훑을 수 있습니다.

왼쪽 액센트 바와 숫자 중심 타이포로 상승/하락 상태를 즉시 식별합니다.
상단 마켓 스트립으로 개별 종목과 시장 온도를 동시에 읽습니다.
실시간 감각은 주되, 레이아웃은 불필요하게 흔들리지 않게 유지합니다.
Primary Value한 눈에 많이 보는 화면. 정보 밀도가 핵심입니다.
UI Motif행, 스파크라인, 얇은 구분선, 경고 pip.
Build HookuseFocusEffect와 5초 폴링.

Build Notes

웹과 모바일이 달라지는 지점만 선명하게

중요한 것은 화려한 차이보다 상태 관리, 폴링, 저장소, 알림처럼 경험을 직접 바꾸는 기술 선택입니다.

Storage

localStorage에서 AsyncStorage로

웹의 동기 저장 대신 React Native 환경에 맞는 비동기 저장을 사용합니다. 키 prefix는 sm_ 체계를 유지합니다.

Notification

브라우저 알림 대신 Expo 로컬 알림

MVP 단계에서는 원격 푸시보다 로컬 알림으로 충분합니다. 권한 요청과 스케줄링 방식이 웹과 다릅니다.

Polling

setInterval보다 포커스 기반 갱신

앱이 보이지 않을 때는 폴링을 멈춰야 합니다. 활성 화면에서만 갱신하는 방식이 배터리와 안정성에 유리합니다.

Workflow

팀이 바로 움직일 수 있는 실행 시퀀스

긴 단계 나열 대신 실제 제작 순서가 읽히는 구조로 재정리했습니다. 아래 단계는 접어서 읽을 수 있습니다.

종목 행과 선물 카드의 타입·색·정보 위계가 맞으면 이후 CRUD와 알림 흐름을 얹기 쉬워집니다.

AsyncStorage keys: sm_watchlist sm_alerts sm_history Notifications: requestPermissionsAsync() scheduleNotificationAsync()

이 단계가 실질적 MVP입니다. 사용자가 앱을 계속 열어두지 않아도 반응을 받을 수 있어야 합니다.

History log payload: { ticker, price, operator, target, time } Debug format: Step 번호 Metro / Expo 에러 전문 실행 환경 재현 방식

제품이 어디서 실패하는지 정리되어야 팀이 빠르게 이어받아 수정할 수 있습니다.

Encoding Guardrail

UTF-8을 끝까지 유지

현재 원본 파일은 일부 환경에서 한글이 깨질 수 있으므로, 새 페이지는 UTF-8 기준으로 저장하고 배포 경로에서도 인코딩이 섞이지 않게 관리해야 합니다.

Embedding Guardrail

iframe 실패를 전제로 읽히게 설계

라이브 프리뷰는 보조 수단입니다. 이 페이지 자체가 독립적인 설명력을 가져야 하므로 링크와 텍스트 설명을 함께 제공합니다.

Next Move

이제 이 컨셉을 실제 웹 또는 앱 작업의 기준 화면으로 쓸 수 있습니다.

랜딩처럼 보이지만, 동시에 제품 브리프와 팀 온보딩 문서 역할도 하도록 구성했습니다. 다음 단계는 이 페이지를 기준으로 실제 앱 구현 우선순위나 웹 소개 페이지 방향을 더 구체화하는 것입니다.
